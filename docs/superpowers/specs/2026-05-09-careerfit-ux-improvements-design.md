# CareerFit Agent UX 改进设计

日期：2026-05-09
版本：v1
范围：5 个 UX 改进方向，分 3 批实施

---

## 1. 设计决策汇总

| 方向 | 选择方案 | 核心思路 |
|------|---------|---------|
| E 空状态与首次体验 | B 分步引导向导 | 全屏向导带用户逐步粘贴 JD → 粘贴简历 → 自动分析 |
| D 输入体验 | A 粘贴即解析 + 结构化预览 | 粘贴后实时提取结构化信息并展示预览卡片 |
| B 报告页信息密度 | C 仪表盘式卡片网格 | 2-3 列卡片网格排列，每张卡片可点击展开详情 |
| C 核心流程连贯性 | A NBA 驱动的流程引导 | 强化 NBA 为全站流程引擎，每步推荐下一步 |
| A 导航精简 | A 合并面试入口 + 分组强化 | 3 个面试入口合并为 1 个，内部 Tab 切换 |

---

## 2. 实施批次与依赖

### 第 1 批：E + D（可并行）

E 和 D 互不依赖，但自然衔接——向导的每一步嵌入解析预览。

### 第 2 批：B + C（强依赖）

仪表盘报告页是 NBA 流程引导的载体——每张卡片的 CTA 就是 NBA。必须先完成报告页重构，才能设计 NBA 跳转逻辑。

### 第 3 批：A（收尾）

流程确定后导航才定型。合并面试入口的 Tab 结构需要与 NBA 跳转协议对齐。

---

## 3. 第 1 批详细设计

### 3.1 E 分步引导向导

**触发条件：** 工作台检测到用户无岗位且无简历时，自动展示向导入口（替代当前 OnboardingGuide 的 3 步文字引导）。

**向导流程：**

```
步骤 1：粘贴岗位描述
  → 文本框粘贴 JD
  → 实时调用解析预览 API
  → 展示解析预览卡片（岗位族、技能维度、要求等级）
  → 用户确认或编辑后进入下一步

步骤 2：粘贴简历
  → 文本框粘贴简历
  → 实时调用解析预览 API
  → 展示解析预览卡片（技能列表、项目经历、教育背景）
  → 用户确认或编辑后进入下一步

步骤 3：开始分析
  → 选择分析模式（完整分析 / 快速分析）
  → 自动创建岗位 + 简历 + 分析任务
  → 显示分析进度
  → 完成后跳转报告页
```

**向导组件：** `OnboardingWizard.vue`，全屏模态，3 步进度条，每步有"上一步"/"下一步"按钮，第 3 步为"开始分析"。

**与现有代码的关系：**
- 替代 `OnboardingGuide.vue`（保留组件但不再作为默认引导）
- 复用 `JobSelector.vue` 和 `ResumeSelector.vue` 的创建逻辑
- 复用 `analyses` store 的 `runAnalysis` action

**跳过机制：** 向导右上角"跳过"按钮，关闭后不再自动弹出（localStorage 记录 `wizard_dismissed: true`）。

### 3.2 D 粘贴即解析 + 结构化预览

**后端新增 API：**

```
POST /api/jobs/parse-preview
  请求体：{ "content": "JD 原文" }
  响应体：{
    "title": "提取的岗位名称",
    "category": "岗位族",
    "skills": [
      { "name": "Python", "level": "required", "category": "technical" },
      { "name": "SQL", "level": "preferred", "category": "technical" }
    ],
    "requirements": ["3年经验", "本科及以上"],
    "domain_keywords": ["后端开发", "分布式系统"]
  }

POST /api/resumes/parse-preview
  请求体：{ "content": "简历原文" }
  响应体：{
    "name": "提取的姓名",
    "skills": ["Python", "SQL", "Docker"],
    "projects": [
      { "name": "项目名", "role": "角色", "highlights": ["亮点1"] }
    ],
    "education": [{ "school": "学校", "major": "专业", "degree": "本科" }],
    "experience_years": 2
  }
```

**前端新增组件：**

- `JdPreviewCard.vue`：展示 JD 解析结果，技能标签可编辑，要求列表可增删
- `ResumePreviewCard.vue`：展示简历解析结果，技能/项目/教育可编辑
- 两个组件都嵌入向导步骤和岗位/简历创建表单中

**解析时机：** 文本框输入停止 800ms 后自动触发（debounce），解析中显示骨架屏，解析失败显示"解析未成功，请直接提交原文"。

**隐私约束：** 解析预览 API 不存储原始文本到日志，不写入数据库。仅返回结构化预览，用户确认后才通过正式 API 创建。

## 4. 第 2 批详细设计

### 4.1 B 仪表盘式卡片网格

**报告页布局：**

```
┌─────────────────────────────────────────────────┐
│  返回工作台  │  报告标题  │  导出按钮  │  NBA    │
├─────────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │ 总分卡片  │  │ 雷达图卡  │  │ NBA行动卡 │      │
│  │ 72/100   │  │ (ECharts) │  │ 练习SQL   │      │
│  └──────────┘  └──────────┘  └──────────┘      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │ 缺口列表卡 │  │ 建议摘要卡 │  │ 面试题卡  │      │
│  │ SQL(高)  │  │ 优化表达  │  │ 3道推荐   │      │
│  │ Docker(中)│  │ 增强证据  │  │          │      │
│  └──────────┘  └──────────┘  └──────────┘      │
│  ┌─────────────────────────────────────────┐    │
│  │ 证据链详情（可折叠）                       │    │
│  └─────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────┐    │
│  │ Agent Trace（可折叠）                     │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

**卡片设计规范：**

- 每张卡片有标题、摘要内容、CTA 按钮
- 点击 CTA 展开详情模态框（非整页跳转）
- 风险信息色 + 文字双通道（如"高风险"红色标签 + "SQL 技能缺失"文字说明）
- 桌面端 3 列，平板 2 列，移动端 1 列

**卡片清单：**

| 卡片 | 内容 | CTA |
|------|------|-----|
| 总分卡片 | 总分大数字 + 分项评分条 | 展开评分详情 |
| 雷达图卡 | SkillsRadarChart | 展开全屏雷达图 |
| NBA 行动卡 | NextBestActionCallout | 执行推荐行动 |
| 缺口列表卡 | 按 priority 排序的 gap 列表 | 展开缺口详情 |
| 建议摘要卡 | 简历优化建议 top 3 | 展开完整建议 |
| 面试题卡 | 推荐 3 道面试题 | 开始面试训练 |

**与现有代码的关系：**
- 复用 `SkillsRadarChart.vue`、`NextBestActionCallout.vue`、`AgentTraceTimeline.vue`
- 重构 `ReportView.vue` 的布局逻辑
- 新增 `ReportCard.vue` 通用卡片容器组件

### 4.2 C NBA 驱动的流程引导

**NBA 跳转协议：**

每个 NBA 推荐包含 `action_type` 和 `target`：

```typescript
interface NBAAction {
  action_type: 'start_interview' | 'start_learning' | 'optimize_resume' | 'create_new_version' | 're_analyze';
  label: string;           // 如"练习 SQL 面试题"
  description: string;     // 如"你的 SQL 技能评分较低，建议通过面试训练提升"
  target_route: string;    // 如'/interview?tab=training&report_id=xxx'
  context: {
    report_id: string;
    job_id: string;
    resume_id: string;
    skill_name?: string;   // 针对的具体技能
  };
}
```

**各页面 NBA 集成：**

| 页面 | NBA 位置 | 可能的推荐 |
|------|---------|-----------|
| 报告页 | NBA 行动卡 | 面试训练/学习任务/简历优化 |
| 面试训练完成 | 页面底部横幅 | 学习任务/新简历版本/再分析 |
| 学习任务完成 | 页面底部横幅 | 新简历版本/再分析 |
| 简历版本创建 | 页面底部横幅 | 再分析 |

**跨页面上下文传递：**
- 使用 URL query 参数传递 `report_id`、`job_id`、`tab`
- 目标页面从 URL 读取参数，自动加载对应上下文
- 不在 localStorage 存储 PII，不使用 URL 传递原文

**面包屑：** 每个页面顶部增加面包屑（工作台 > 报告 > 面试训练），解决"报告页无返回入口"问题。

## 5. 第 3 批详细设计

### 5.1 A 合并面试入口 + 分组强化

**侧边栏变更：**

```
当前（9 项）：                    改进后（7 项）：
├ 工作台                         ├ 工作台
├ 岗位管理                       ├ 岗位管理
├ 简历管理                       ├ 简历管理
├ 历史趋势                       ├ ── 洞察 ──
├ 版本对比                       ├ 历史趋势
├ 面试准备（学习任务）            ├ 版本对比
├ 面试训练（模拟面试）   →       ├ 面试（Tab: 学习任务 | 模拟面试 | 题目生成）
├ 面试题库（题目生成）            ├ ── 系统 ──
├ 设置                           ├ 设置
```

**面试页 Tab 结构：**

```
/interview?tab=learning     → 学习任务（原 LearningTasksView）
/interview?tab=training     → 模拟面试（原 InterviewDetailView）
/interview?tab=bank         → 题目生成（原 InterviewListView 的题库功能）
```

**NBA 跳转集成：** NBA 的 `target_route` 使用 `?tab=` 参数自动激活对应 Tab，如 `/interview?tab=training&report_id=xxx`。

**路由变更：**
- 新增 `/interview` 路由（InterviewView.vue，Tab 容器）
- 保留 `/interview-prep`、`/interview-training`、`/interview-bank` 的重定向到 `/interview?tab=xxx`
- 侧边栏 `SideNav.vue` 更新导航项

## 6. 设计约束

- 不引入登录/注册/多租户
- 不使用 mock 数据假装功能存在
- 前端 localStorage 不存储简历原文、JD 原文、Agent trace 原文等 PII
- 跨页面数据传递使用 ID 引用（`report_id`、`job_id`），不传递原文
- 移动端必须可用：仪表盘降级为单列，向导全屏适配
- 解析预览 API 不存储原始文本到日志，不写入数据库
- 风险信息必须色 + 文字双通道
- 所有按钮/输入框必须有完整状态（默认/hover/focus/active/disabled/loading/error）
