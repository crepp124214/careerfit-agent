# CareerFit 项目性能优化方案

## 📊 项目概览

**项目架构**: FastAPI + Vue.js + PostgreSQL/pgvector + LangGraph
**核心功能**: 职位分析、简历匹配、LLM 工作流、RAG 检索
**性能目标**: 支持并发分析任务，响应时间 < 3s，吞吐量 > 10 TPS

---

## 🔍 性能问题汇总

### 高优先级问题（立即修复）

#### 1. N+1 查询问题
- **位置**: `analysis_service.py` - `_build_rag_results`
- **问题**: 对每个技能循环调用 `retrieve_by_skill`，导致多次数据库查询
- **影响**: 技能数量越多，查询次数线性增长
- **优化方案**: 
  ```python
  # 当前实现（问题）
  for skill in required_skills:
      documents = retrieve_by_skill(db, skill, top_k=3)  # N次查询
  
  # 优化方案
  documents = retrieve_by_skills_batch(db, required_skills, top_k=3)  # 1次查询
  ```

#### 2. RAG 检索性能瓶颈
- **位置**: `retrieval.py` - `_retrieve_json_fallback`
- **问题**: 遍历所有文档计算相似度，O(n) 复杂度
- **影响**: 文档数量增长时性能急剧下降
- **优化方案**:
  - 使用 pgvector 的向量索引（IVFFlat 或 HNSW）
  - 实现批量相似度计算
  - 添加缓存层

#### 3. 数据库连接管理
- **位置**: `analysis_service.py` - `_run_analysis_background`
- **问题**: 后台线程创建独立会话，可能导致连接泄漏
- **影响**: 并发任务多时连接池耗尽
- **优化方案**:
  ```python
  # 使用上下文管理器确保会话关闭
  with SessionLocal() as db:
      try:
          _execute_analysis_core(...)
      finally:
          db.close()
  ```

### 中优先级问题（计划修复）

#### 4. LLM 调用优化
- **位置**: `agents/graph.py` - LLM 节点
- **问题**: 缺乏有效的并发控制和缓存
- **优化方案**:
  - 实现请求队列和限流
  - 添加响应缓存（相同 prompt 直接返回缓存结果）
  - 优化 prompt 模板减少 token 使用

#### 5. 事件回放无缓存
- **位置**: `api/routes/analysis.py` - `_replay_events_from_runs`
- **问题**: 每次重放事件都查询数据库
- **优化方案**: 添加 Redis 缓存层

#### 6. 向量计算优化
- **位置**: `retrieval.py` - `_cosine_similarity`
- **问题**: 未使用向量归一化，计算效率低
- **优化方案**: 预计算归一化向量，使用矩阵运算

## 🏗️ 架构优化建议

### 1. 引入缓存层
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Client    │────▶│  API Layer  │────▶│    Cache    │
└─────────────┘     └─────────────┘     │   (Redis)   │
                                        └──────┬──────┘
                                               │
                                        ┌──────▼──────┐
                                        │  Database   │
                                        └─────────────┘
```

**缓存策略**:
- LLM 响应缓存（TTL: 1小时）
- RAG 检索结果缓存（TTL: 30分钟）
- 分析报告缓存（TTL: 24小时）
- 事件回放缓存（TTL: 10分钟）

### 2. 异步任务队列
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Client    │────▶│   FastAPI   │────▶│    Redis    │
└─────────────┘     └─────────────┘     │    Queue    │
                                        └──────┬──────┘
                                               │
                                        ┌──────▼──────┐
                                        │   Celery    │
                                        │   Worker    │
                                        └─────────────┘
```

**适用场景**:
- 分析任务执行
- 报告生成
- 批量数据处理

### 3. 数据库优化
- **连接池配置**: 
  ```python
  engine = create_engine(
      DATABASE_URL,
      pool_size=20,
      max_overflow=30,
      pool_pre_ping=True,
      pool_recycle=3600
  )
  ```
- **索引优化**:
  - 为 `AnalysisTask.status` 添加索引
  - 为 `AgentRun.task_id` 添加索引
  - 优化 pgvector 向量索引（HNSW）

## 📈 性能测试方案

### 测试环境
- **硬件**: 4核8G 服务器
- **数据库**: PostgreSQL 14 + pgvector
- **并发用户**: 10-100 虚拟用户

### 测试场景

#### 场景 1: 单用户基准测试
- **目标**: 建立性能基线
- **用例**: 
  - 创建分析任务
  - 查询分析结果
  - 获取报告
- **预期指标**:
  - 响应时间 < 2s
  - 错误率 < 1%

#### 场景 2: 并发分析任务测试
- **目标**: 测试系统并发处理能力
- **用例**: 10/50/100 用户同时创建分析任务
- **预期指标**:
  - 平均响应时间 < 5s
  - 吞吐量 > 10 TPS
  - 错误率 < 5%

#### 场景 3: RAG 检索性能测试
- **目标**: 测试检索模块性能
- **用例**: 
  - 单技能检索
  - 多技能批量检索
  - 大数据量检索
- **预期指标**:
  - 检索时间 < 500ms
  - 内存使用 < 500MB

#### 场景 4: LLM 调用性能测试
- **目标**: 测试 LLM 模块性能
- **用例**:
  - 单 prompt 调用
  - 并发 prompt 调用
  - 缓存命中率测试
- **预期指标**:
  - 响应时间 < 3s
  - 缓存命中率 > 30%

### 测试工具
- **Locust**: API 负载测试
- **pytest-benchmark**: 单元性能测试
- **pgbench**: 数据库性能测试
- **Prometheus + Grafana**: 性能监控
